Zero-Shot Prompting

모델에게 예시를 전혀 제공하지 않고 지시만으로 태스크를 수행시키는 가장 기본적인 기법입니다. LLM의 사전 학습 지식에 전적으로 의존합니다.

Technique
PROMPT > 다음 영화 리뷰의 감정을 긍정/부정으로 분류하시오. REVIEW > "이 영화는 시간 낭비였다. 스토리가 너무 진부했다." OUTPUT > 부정 (Negative)
Level: Beginner
NLP Classification Sentiment
Advanced

Chain-of-Thought

"Step by step으로 생각하시오"라는 지시를 포함하여, 모델이 최종 답변을 내리기 전 중간 추론 과정을 명시적으로 생성하도록 유도합니다. 복잡한 수학 문제나 다단계 논리 문제에 탁월한 성능을 발휘합니다.

단계별로 추론하시오. Q: 사과 5개에서 2개를 먹고, 친구에게 1개를 주면? A: 1단계: 5 - 2 = 3개 남음 2단계: 3 - 1 = 2개 남음 최종 답: 2개
정확도 향상
복잡도 대응
설명 가능성

Role Prompting (Persona)

모델에게 특정 전문가 역할이나 페르소나를 부여하여 해당 관점에서 응답하도록 유도합니다. "당신은 10년 경력의 보안 전문가입니다"와 같은 시스템 프롬프트가 대표적입니다.

Technique
SYSTEM: 당신은 15년 경력의 사이버 보안 전문가입니다. 기술적 정확성과 명확한 설명을 최우선으로 합니다. USER : SQL Injection 공격 원리를 초보자에게 설명해 주세요.

FAQ

자주 묻는 질문들

프롬프트 엔지니어링을 처음 접할 때 가장 많이 하는 질문들을 모았습니다.

Q 프롬프트 엔지니어링은 개발자만 배울 수 있나요?

전혀 그렇지 않습니다. 프롬프트 엔지니어링은 코딩 지식이 없어도 배울 수 있는 커뮤니케이션 기술에 가깝습니다. 마케터, 작가, 연구자, 교육자 등 다양한 직군의 전문가들이 이미 적극적으로 활용하고 있습니다. 핵심은 AI 모델이 어떻게 생각하는지를 이해하고, 그에 맞게 의도를 명확히 전달하는 능력입니다.

  • 비개발자 친화적
  • 커뮤니케이션 스킬
  • 진입 장벽 낮음
Q GPT, Claude, Gemini 등 모델마다 프롬프트가 달라야 하나요?

핵심 원칙(Context, Task, Constraint, Format)은 어느 모델에나 동일하게 적용됩니다. 다만 모델별 특성과 학습 방식이 다르기 때문에 세부 표현이나 강조점은 조금씩 달라질 수 있습니다. 예를 들어 Claude는 역할 부여에 민감하고, GPT-4o는 단계별 지시에 잘 반응하는 경향이 있습니다.

  • 모델 불가지론적 접근
  • 미세 조정 필요
Q Few-Shot 프롬프트는 언제 사용해야 하나요?

출력 형식이 매우 특정하거나, Zero-Shot으로 원하는 결과를 얻기 어려울 때 사용합니다. 예를 들어 특정 브랜드 보이스로 카피라이팅을 해야 하거나, 정형화된 JSON/CSV 구조로 데이터를 추출해야 할 때 2~5개의 예시를 제공하면 정확도가 크게 향상됩니다. 단, 예시가 너무 많으면 토큰 비용이 증가하므로 균형을 맞추는 것이 중요합니다.

  • Few-Shot
  • 2-5개 예시 권장
  • 구조화된 출력
Q Chain-of-Thought(CoT)는 항상 효과적인가요?

CoT는 추론 단계가 필요한 복잡한 문제에서 뛰어난 효과를 보입니다. 수학 계산, 논리적 추론, 다단계 분석 등이 대표적입니다. 반면 간단한 사실 확인이나 단순 번역처럼 직접적인 답이 있는 작업에는 불필요한 토큰 낭비가 될 수 있습니다. “단계적으로 생각해 줘(Think step by step)”라는 한 문장이 놀라운 성능 향상을 가져올 수 있습니다.

  • Chain-of-Thought
  • 복잡한 추론에 최적
  • 토큰 효율 고려
Q 프롬프트가 길수록 결과가 좋아지나요?

길이와 품질은 비례하지 않습니다. 명확성과 밀도가 핵심입니다. 불필요한 수식어, 중복 지시, 모호한 표현은 오히려 모델을 혼란스럽게 합니다. “친절하고 전문적이고 간결하고 이해하기 쉽게” 같은 중첩된 수식보다 “대상 독자: 비전공자 성인, 톤: 간결하고 친근하게” 처럼 구조화된 지시가 훨씬 효과적입니다.

  • 명확성 > 길이
  • 구조화 우선
  • 중복 제거
Q System Prompt와 User Prompt의 차이는 무엇인가요?

System Prompt는 AI의 전체 행동 방식, 역할, 제약 조건을 정의하는 고수준 지시입니다. 마치 직원 교육 매뉴얼처럼, 모든 대화에 일관되게 적용됩니다. 반면 User Prompt는 개별 대화에서의 구체적인 요청입니다. API를 직접 활용하는 개발자라면 System Prompt를 통해 페르소나, 응답 형식, 금지 사항을 명확히 설정하는 것이 매우 중요합니다.

  • System Prompt
  • User Prompt
  • 역할 분리 설계

Common Pitfalls

자주 하는 실수들

카드 위에 마우스를 올려 실수의 해결책을 확인하세요. 작은 습관의 차이가 프롬프트 품질을 결정합니다.

Mistake 01

너무 모호한 지시

"좋은 글 써줘", "코드 고쳐줘"처럼
목적·대상·형식이 전혀 없는
지시는 엉뚱한 결과를 냅니다.

뒤집어서 해결책 보기
Solution

Context + Task 명시

"마케터를 위한 SNS 캡션을
인스타그램 스타일로, 150자 이내로
3가지 버전으로 작성해 줘."

Role + Goal + Format + Limit
Mistake 02

역할 부여 없음

AI에게 '누구처럼' 행동할지
알려주지 않으면 일반적이고
밋밋한 답변만 돌아옵니다.

뒤집어서 해결책 보기
Solution

Persona 설정

"당신은 10년 경력의 UX 라이터입니다.
전환율 최적화 관점에서
랜딩 페이지 카피를 작성해 주세요."

Act as [Expert] with [Experience]
Mistake 03

출력 형식 미지정

결과물의 형식(JSON, 표, 목록 등)을
지정하지 않으면 매번
다른 형태로 응답이 옵니다.

뒤집어서 해결책 보기
Solution

Format 명시

"결과는 JSON 형식으로,
title / summary / tags
3개의 키만 포함해 주세요."

Output: JSON / Markdown / Table
Mistake 04

한 번에 너무 많은 요청

하나의 프롬프트에 10가지를
한꺼번에 요청하면 AI가
우선순위를 잃고 품질이 저하됩니다.

뒤집어서 해결책 보기
Solution

단계별 Chain 분리

작업을 3~4단계로 쪼개어
순서대로 요청하는
Chain-of-Thought 전략을 사용하세요.

Step 1 → Step 2 → Step 3
